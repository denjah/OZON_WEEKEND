# ТЗ: Сервис SEO-аудита карточек Ozon через API

**Заказчик:** магазин-игротека (настольный теннис, аэрохоккей, батуты, баскетбол, снукер и комплектующие)
**Назначение:** автоматизировать ручную методику SEO-сверки, описанную во вкладке «Инструкция» рабочего файла (`ozon_keywords`). Сервис должен повторять цепочку **Ключи → Сверка карточки → Конкуренты → Итог**, но без ручного труда.
**Версия:** 1.0 (черновик для оценки)

---

## 1. Цель

Для каждого товара магазина сервис проверяет, насколько наименование, характеристики и описание соответствуют реальным поисковым запросам покупателей, сравнивает покрытие с конкурентами и выдаёт конкретные рекомендации: какой ключ и куда (название / характеристики / описание) добавить.

Логика повторяет ручной процесс из файла:

1. Собрать целевые ключи и их частотность (лист «Ключи»).
2. Пословно сверить карточку с ключами (лист «Сверка карточки»).
3. Сравнить покрытие с конкурентами и найти разрывы (лист «Конкуренты»).
4. Свести покрытие и приоритеты (лист «Итог»).

---

## 2. Входные данные и доступы

- **Seller API Ozon.** Авторизация заголовками `Client-Id` и `Api-Key`, база `https://api-seller.ozon.ru`. Ключ выдаёт заказчик из личного кабинета.
- **Подписка.** Часть аналитики по запросам шире на тарифах Premium / Premium Plus (больший период и объём данных). Уточнить у заказчика текущий тариф — от этого зависит глубина истории.
- **Словари ниш** (конфиг, ведёт SEO-специалист, не хардкод) — см. раздел 6.
- **Сиды по нишам** (конфиг) — список высокочастотных затравочных фраз на каждую нишу.

---

## 3. Источники данных по API

| Что нужно | Метод | Что отдаёт |
|---|---|---|
| Список товаров магазина | `POST /v3/product/list` | `product_id`, `offer_id`, `sku` |
| Базовая карточка | `POST /v3/product/info/list` | категория, статус, штрихкоды |
| Наименование и характеристики | `POST /v4/product/info/attributes` | `name`, массив `attributes` (все атрибуты карточки) |
| Описание | `POST /v1/product/info/description` | `name` + `description` (rich-контент) |
| Запросы своих товаров | `POST /v1/analytics/product-queries` | список запросов, по которым показываются товары магазина, с метриками |
| Детализация по запросу | `POST /v1/analytics/product-queries/details` | метрики по конкретным запросам |
| Общая аналитика (показы/заказы) | `POST /v1/analytics/data` | метрики по `dimensions` (для приоритизации товаров) |

## 4. Критичные ограничения (заложить в архитектуру сразу)

**4.1. Частотность произвольных ключей.** В кабинете есть seed-поиск: вводишь любую фразу — получаешь до 1000 связанных запросов с популярностью. **Прямого публичного API-аналога этому нет.** `product-queries` отдаёт только запросы, по которым уже показываются товары магазина. Поэтому для добора высокочастотных сидов, которых в карточке ещё нет, предусмотреть один из вариантов (согласовать с заказчиком):

- **Вариант A (MVP):** на вход подаётся ручная выгрузка CSV из раздела «Поисковые запросы» по списку сидов. Сервис парсит её как справочник частотностей. Минус — ручной шаг.
- **Вариант B:** интеграция со сторонним поставщиком базы частотностей (MPSTATS, SellerStats, Moneyplace) по их API. Минус — платно, свериться с их покрытием по нужным категориям.
- **Вариант C:** работать только по `product-queries` своих товаров. Минус — не увидим ключи, по которым товар вообще не показывается (а это главный источник роста).

Рекомендация: **A на MVP, B на втором этапе.**

**4.2. Карточки конкурентов.** Seller API отдаёт только товары своего магазина. Наименование / характеристики / описание конкурента берутся либо парсингом публичной страницы товара (`ozon.ru/product/...`), либо через сторонний API. Парсинг публичных страниц согласовать с заказчиком на предмет допустимости и устойчивости (антибот, смена вёрстки). Список конкурентов на товар задаёт SEO-специалист (URL или артикулы) — авто-подбор конкурентов в MVP не делаем.

**4.3. Лимиты запросов.** Соблюдать rate limits Seller API (батчи по товарам, ретраи с backoff, очередь). Точные лимиты взять из текущей документации dev.ozon.ru на момент разработки.

## 5. Функциональный конвейер

### Этап 1. Инвентаризация магазина и Слой 0 (эталон из кабинета)
Выгрузить все товары (`product/list` → `info/attributes` + `info/description`). Для каждого сохранить: `offer_id`, `name`, плоский список атрибутов (ключ-значение), `description`. Это **Слой 0 — эталон**, с которым сверяется всё дальнейшее.

**Расшифровка характеристик (обязательный шаг).** `info/attributes` возвращает характеристики идентификаторами (`attribute_id`, `dictionary_value_id`), а не текстом. Чтобы получить читаемые «Назначение = Для помещений», нужен справочник категории: `/v1/description-category/tree` (→ `type_id`) → `/v1/description-category/attribute` (расшифровка `attribute_id`) → `/v1/description-category/attribute/values` (расшифровка значений). Справочник категории меняется редко — кешировать надолго; значения конкретных товаров тянуть при каждом аудите.

**Классификация ниши — по атрибуту «Тип», а не по словам в названии.** Это критичное правило. «Балка для батута 366 см» содержит слово «батут», но её Тип — запчасть; классифицировать по слову в заголовке нельзя, иначе запчасть попадёт в нишу батутов. Определяем нишу из атрибута «Тип» (и смежных «Назначение», «Вид»), затем сверяем со справочником ниш (раздел 6).

**Три типа карточек** (от этого ветвится вся логика):

| Тип карточки | Как определяем нишу | Целевые ключи |
|---|---|---|
| Одиночная игра | один Тип из атрибутов | ключи этой ниши минус конфликты атрибутов |
| Комплектующее | свой Тип (запчасть/аксессуар) | деталь + бренд + размер; без широких ключей родителя |
| Комбо (X в 1) | набор игр, извлечённый из описания | первичная ниша — детские игровые столы; см. Этап 3 |

### Этап 2. Сбор ключей по нише
Для ниши товара взять сиды из конфига, подтянуть частотности (источник по разделу 4.1). Сформировать пул ключей с полями, как в листе «Ключи»: запрос, частотность, число конкурентов (если доступно), приоритет.

### Этап 3. Фильтрация пула (по требованиям заказчика)
- **Отсечь частотность < 10.** Запросы с популярностью ниже 10 в анализ не берём.
- **Отсечь чужую нишу — по конфликту с атрибутами Слоя 0.** Запрос нецелевой, если его слова противоречат атрибутам товара. Примеры: атрибут «Назначение = Для помещений» → конфликт со словами уличный, всепогодный, для дачи; «Размер = 274» (полноразмерный) → конфликт с мини, детский; «Тип = Теннисный стол» → конфликт с ракетка, чехол (это комплектующие). Дополняется ручным словарём стоп-слов ниши (бренды конкурентов и т.п.).
- Логировать, что и почему исключено (для прозрачности, как серые строки в «Ключах»).

**Комбо-столы (X в 1) — особый случай, правило «одна карточка = одна ниша» не применяется.** Стол 13-в-1 / 12-в-1 / 4-в-1 законно принадлежит многим нишам сразу. Логика:

- Нейросеть **извлекает** из описания список входящих игр и их размеры полей (текст структурирован: каждая игра через «☆» или маркер, с комплектацией). ИИ только парсит список — решение «целевой/нет» остаётся за правилами.
- **Первичная ниша комбо — детские игровые столы.** Главный кластер: «игровой стол X в 1», «детский игровой стол многофункциональный», «стол трансформер игровой». Отдельные игры внутри — вторичный длинный хвост.
- Ключ целевой, если: совпал с любой входящей игрой **И** не противоречит размеру/качеству карточки. Пример: в 12-в-1 теннисное поле 101×53 см — мини, поэтому «профессиональный теннисный стол 274» нецелевой, а «мини теннисный стол детский» целевой.
- Исключение по умолчанию: «детский» снимается, только если атрибуты товара явно говорят о взрослом/семейном уровне (хромированные штанги, кии и шары, полноразмерные поля) — тогда комбо повышается в «семейный/взрослый игровой стол». Решение фиксируется на уровне SKU.

### Этап 4. Пословная сверка карточки
Для каждого оставшегося ключа проверить присутствие его значимых слов (после нормализации — лемматизация, нижний регистр, без стоп-слов языка) в трёх полях по отдельности: **наименование**, **характеристики**, **описание**. Считать статус по логике листа «Сверка»:

- покрытие = число полей из трёх, где ключ присутствует;
- статус: `Отсутствует` (0 полей) → `Нет в названии` (есть, но не в name) → `Частичное` → `Полное` (все три).

### Этап 5. Сравнение с конкурентами
По тем же ключам проверить присутствие в карточках конкурентов (Да / Частично / Нет). Вычислить **Разрыв**: ключ есть у конкурента и отсутствует/частичен у нас. Это прямой аналог колонки «Разрыв».

### Этап 6. Рекомендации
Для каждого ключа со статусом хуже «Полное» и частотностью ≥ 10 предложить **куда вписать**, по правилу приоритета:

- высокочастотный и отсутствует в наименовании → кандидат в **наименование** (с учётом лимита длины Ozon и читаемости);
- среднечастотный или уточняющий (материал, размер, толщина, комплектация) → **характеристики**;
- остальное и развёрнутые формулировки → **описание**.

Сгенерировать черновые предложения по трём полям: новое наименование (с добором недостающих ВЧ-ключей и комплектации), список атрибутов на дозаполнение, тезисы для описания. Не дублировать чужие бренды, не вписывать ключи чужой ниши (см. флажки в «Ключах»).

## 6. Классификация и справочник ниш (конфиг)

Справочник — отдельный редактируемый файл (`Справочник_ниш_ozon.xlsx`, прилагается), ведёт SEO-специалист, сервис только применяет. Структура на нишу: `категория`, `тип карточки`, `эталонный атрибут «Тип»`, `сиды` (затравки для частотностей), `целевые_маркеры`, `стоп_слова` (чужая ниша), `связь/родитель`. Лист «Правила классификации» в том же файле фиксирует логику ветвления.

Главные правила (повторяются в файле):

- **Классифицируем по атрибуту «Тип» из карточки, а не по слову в названии.** Запчасть с «батут» в заголовке не уходит в нишу батутов.
- **Кросс-нишевое исключение.** Ключ, противоречащий атрибутам товара или попадающий под стоп-слова ниши, не анализируется и не рекомендуется.
- **Комплектующие — самостоятельные ниши.** Целевые ключи про саму деталь + бренд + размер (совместимость), без широких ключей родителя.
- **Хоккей раздвоен:** рычажный (штанговый, с фигурками) и аэрохоккей (на воздушной подушке) — разные ниши со взаимными стоп-словами.
- **Теннис раздвоен:** indoor/профессиональный (для помещений) и outdoor (всепогодный/уличный) — взаимные стоп-слова; плюс отдельная ниша мини/детский.
- **Комбо (X в 1) — первично детские игровые столы** (см. Этап 3), с мульти-меткой по входящим играм.

## 7. Выходные данные

**По товару** (json + выгрузка в xlsx, повторяющая листы файла):
- ключи с частотностью, статусом покрытия и «куда вписать»;
- процент покрытия (как «Итог»);
- список разрывов с конкурентами;
- черновые рекомендации по наименованию / характеристикам / описанию.

**По магазину:** сводная таблица товаров с % покрытия, числом разрывов и количеством рекомендаций; сортировка по приоритету (товары с высоким спросом и низким покрытием — наверх).

## 7.1. Дашборд: метрики и логика вердикта

Задача дашборда — не показать максимум цифр, а ответить на один вопрос по каждой карточке: **нужен SEO-ремонт или проблема в другом (цена/фото/отзывы)?** Сводный пример вёрстки — `dashboard_mockup.html`.

**Верхний уровень — четыре счётчика-статуса:** «Ремонт SEO», «SEO + видимость», «SEO ок — цена/фото», «В норме». Дают мгновенную картину по магазину.

**Блок «Отслеживаемые карточки»** — расположен под счётчиками-статусами и над таблицей «Все карточки». Свёрнут под спойлер, сверху выпадающий фильтр по товару. Структура иерархическая: каждый товар, по которому добавлена хотя бы одна гипотеза, образует **отдельный подраздел (карточку товара)**; все отслеживаемые запросы этого товара лежат внутри его подраздела. Гипотезы по другому товару создают новый подраздел со своей карточкой.

По каждому отслеживаемому запросу внутри подраздела: «Отслеживаемый запрос», «Действие» (переносится из колонки A гипотезы), «Когда добавлен», и пары метрик «на момент добавления → на нынешний момент»: переходы, CTR, конверсия, место по запросу. Плюс кнопка удаления запроса. Снимок метрик фиксируется в момент добавления; «нынешние» подтягиваются при каждом прогоне — так виден эффект правки по конкретному запросу (цикл до/после).

**Таблица «Все карточки»** — все товары магазина, сортировка «хуже сверху». Видимые колонки: Товар, Покрытие ключей (% с прогресс-баром), Разрывы, Показы (динамика), CTR, Конв. в корзину, CPC, Вердикт. Колонки «Соответствие запросу» и «Позиция» в сводную таблицу не выводятся (остаются в детализации карточки — раздел 7.2), чтобы не перегружать список. Полный набор метрик по группам, который считается под капотом:

| Группа | Метрики | Откуда |
|---|---|---|
| Релевантность (SEO-сигнал) | Соответствие запросу; покрытие ключей %; разрывы с конкурентами; индекс видимости | Ozon (промо/аналитика) + наш инструмент |
| Видимость | Показы всего и в поиске + динамика; позиция в поиске/каталоге; доля запросов без действий | Ozon аналитика |
| Вовлечение/конверсия | CTR (показ→клик); конверсия в корзину; конверсия в заказ | Ozon аналитика |
| Экономика рекламы | Ставка / стоимость клика (CPC); доля рекламных показов и продаж; ДРР; «переплата из-за релевантности» | Ozon продвижение |

**Колонка «Вердикт» (светофор) — главный результат, считается из метрик:**

- **Ремонт SEO (красный):** соответствие запросу низкое И покрытие ключей низкое → текст карточки не подходит запросам.
- **SEO + видимость (жёлтый):** соответствие среднее, показы падают, позиция глубокая → добрать ключи и подключить продвижение.
- **SEO ок — цена/фото (синий):** соответствие и покрытие высокие, но низкий CTR/конверсия → текст не трогать, работать с ценой, фото, отзывами.
- **В норме (зелёный):** релевантность, показы и конверсия в порядке.

**Связь релевантности и CPC (заложить в метрику «переплата»).** По единой формуле ранжирования Ozon органическая оценка (релевантность) применяется как коэффициент до рекламного буста: при низком «Соответствии запросу» нужна более высокая ставка за ту же позицию. Метрика показывает, насколько текущий CPC выше минимального по нише при текущей релевантности — переводит SEO-ремонт в деньги и обосновывает приоритет.

**Важно про конкурентов.** «Соответствие запросу» от Ozon — интегральный коэффициент: индикатор «нужен ли ремонт», но он не показывает, какого ключа не хватает. Конкретный список недостающих ключей по-прежнему даёт лист «Сверка». На дашборде — индикатор; в детализации товара — пословные разрывы.

## 7.2. Детализация карточки и гипотезы ремонта (HADI)

**Каждый вердикт на дашборде — кликабельная кнопка.** По клику пользователь проваливается в детализацию товара. Цель экрана — не просто показать проблему, а выдать готовый план ремонта.

Экран детализации состоит из двух частей:

1. **Доказательная база.** Те же находки, на которых система построила вердикт: соответствие запросу, покрытие ключей, список недостающих ключей с частотностью, разрывы с конкурентами, CTR, конверсия, позиция, динамика показов. Пользователь видит, *почему* выставлен именно этот статус.
2. **Гипотезы ремонта в формате HADI**, отсортированные по ожидаемому эффекту.

**Ключевой принцип: гипотезы строятся на той же аналитике, что дала вердикт.** Система хранит по каждому товару набор находок (`findings`) и маппит каждую находку в шаблон гипотезы. Нейросеть формулирует человекочитаемый текст, но данные — ключ, частотность, метрика, разрыв — берутся из анализа, не выдумываются. Это та же граница, что и при классификации: правила дают факты, ИИ формулирует.

**HADI-гипотезы — всегда только по SEO.** Инструмент работает с наименованием, характеристиками и описанием, поэтому развёрнутые гипотезы в формате HADI генерируются исключительно для SEO-правок. Проблемы вне SEO (цена, фото, отзывы) инструмент не планирует детально — он их только обозначает.

**Поведение для не-SEO вердиктов (например «SEO ок — цена/фото»).** Проваливание работает так же, но экран строится иначе:

1. **Доказательная база** — как обычно.
2. **Краткая сводка «Что починить» — 1–2 предложения** о не-SEO проблеме, без HADI. Например: «Текст релевантен (покрытие 88%, позиция 17), но CTR 0,8% при 35 тыс. показов. Чинить главное фото/инфографику и проверить цену против конкурента — текст не трогать».
3. **HADI-гипотезы — только по SEO.** Если по тексту остались мелкие недоборы (хвостовые ключи) — выводятся низким приоритетом. Если SEO в норме — блок показывает «По SEO правок не требуется», без выдуманных гипотез.

То есть для SEO-вердиктов HADI — основное содержимое экрана, для не-SEO — короткая сводка вне SEO плюс HADI только по остаточным текстовым правкам (или их отсутствию).

**Формат HADI — 6 колонок** (повторяет рабочий HADI-лист):

| Колонка | Содержимое |
|---|---|
| Гипотеза (H) | если [действие], то [метрика] вырастет, потому что [находка из анализа] |
| Действие (A) | конкретная правка: название / характеристики / описание (для не-SEO — обозначается в сводке, не в HADI) |
| Данные — что считаем (D) | метрика: показы, позиция, CTR, соответствие запросу, покрытие |
| Чем мерить (инструменты) | **конкретный запрос для отслеживания** и где его смотреть (Ozon → Аналитика → Товары в поиске по этому запросу) |
| Ориентир (целевое изменение) | целевое значение: «соответствие 34→70», «позиция 52→топ-20» |
| Горизонт | срок перепроверки: +14 / +21 день |

Поле «Инсайт» (подтверждена/опровергнута) заполняется после следующего прогона — связано с историей прогонов из раздела 8.

**В каждом провале — 5 гипотез, ранжированных по мэтчу × влиянию на результат.** Мэтч — насколько гипотеза соответствует найденной проблеме (уверенность, что правка её закрывает); влияние — ожидаемый размер эффекта (сумма частотности недостающих ключей, величина разрыва, отклонение метрики от нормы по нише). Если реальных гипотез меньше пяти — показываем меньше, недостающие выдуманными не добиваем.

**Добавление в отслеживание.** У каждой гипотезы — действие «отслеживать»: в блок «Отслеживаемые карточки» переносятся запрос из колонки «Чем мерить», «Действие» (A) этой гипотезы и снимок метрик на момент добавления (переходы, CTR, конверсия, место по запросу). Запрос попадает в подраздел своего товара (если подраздела ещё нет — он создаётся). При следующем прогоне рядом со снимком показываются текущие значения тех же метрик — виден эффект правки.

**Маппинг вердикт → находки → гипотезы:**

| Вердикт | Типовые находки | Что генерируют гипотезы |
|---|---|---|
| Ремонт SEO | низкое соответствие, покрытие < 50%, недостающие ВЧ-ключи | переписать название под главный кластер; дозаполнить характеристики; добрать ключи в описание |
| SEO + видимость | среднее соответствие, падающие показы, глубокая позиция | добрать недостающие ключи + подключить «Оплату за клик» на ВЧ-запросы пока органика догоняет |
| SEO ок — цена/фото | высокое покрытие, но низкий CTR/конверсия | НЕ трогать текст; гипотезы про цену, инфографику, отзывы |
| В норме | отклонений нет | гипотез нет, мониторинг |

**Примеры сгенерированных гипотез:**

- *Ремонт SEO (комбо 12-в-1):* «Если вынести в название "детский игровой стол 12 в 1" и перечень игр, соответствие и показы по детским запросам вырастут — сейчас покрытие 41%, главный кластер не покрыт». A: переписать название. D: соответствие запросу + показы в поиске, +14 дней.
- *SEO + видимость (проф. стол):* «Если добавить в название комплектацию и "для помещений", добрать "25 мм", покрытие вырастет и позиция поднимется с 52». D: позиция, показы, +14 дней.
- *SEO ок — цена/фото (всепогодный лидер):* экран показывает краткую сводку вне SEO — «Текст релевантен (покрытие 88%), CTR 0,8% при показах 35 тыс. Чинить фото/инфографику и цену, текст не трогать» — и блок HADI «По SEO правок не требуется» (или 1 хвостовой ключ низким приоритетом).

**Приоритизация гипотез:** по ожидаемому эффекту — сумма частотности недостающих ключей, величина разрыва с конкурентами, размер отклонения метрики от нормы по нише. Гипотезы выгружаются в HADI-лист (как в рабочем файле), чтобы отслеживать цикл и не терять историю проверок.

## 8. Нефункциональные требования

- Хранение выгрузок и истории прогонов (для динамики «до/после», как HADI-цикл в файле).
- Конфиги ниш и сидов — редактируемые без правки кода (yaml/БД + простой интерфейс или таблица).
- Идемпотентность и ретраи на API; защита `Api-Key` (секрет-хранилище, не в коде).
- Нормализация русского текста: лемматизация (pymorphy/Natasha), снятие регистра, разные порядки слов считать одним ключом по множеству лемм.
- Логи исключённых ключей и причин.

## 9. Этапность

- **MVP:** этапы 1–4 + 6 на одной нише (настольный теннис, indoor), частотности из ручного CSV (вариант A), без конкурентов. Выход — отчёт по товару с рекомендациями.
- **Этап 2:** сравнение с конкурентами (этап 5), парсинг публичных карточек или сторонний API.
- **Этап 3:** автоматический сбор частотностей (вариант B), остальные ниши, сводка по магазину и динамика прогонов.
